# 共享 Planning & Check-in 规范

本规范适用于仓库内所有 skills。

目标只有一个：任何技能在执行时，都不能只“开始干活”，而必须先说明计划、在关键阶段停下来回报，并在必要时等待确认后再继续。

## 1. 强制执行节奏

每个技能都必须遵循下面的节奏：

1. `计划确认`
2. `阶段确认`
3. `偏差确认`
4. `收尾确认`

### 1.1 计划确认

在任何脚本执行、批量分析、轮次迭代或大段结论输出前，必须先给出一次 `计划确认`，至少包含：

- 当前目标
- 已知输入
- 缺失信息或默认假设
- 预期产物
- 执行步骤
- 关键风险
- 是否需要用户确认

推荐格式：

```text
【计划确认】
- 当前目标：
- 已知输入：
- 缺失信息 / 默认假设：
- 预期产物：
- 执行步骤：
- 风险与停止条件：
- 是否需要你确认：
```

### 1.2 阶段确认

每进入一个新阶段，或准备从“轻量试探”切到“正式执行”时，必须给出一次 `阶段确认`，至少包含：

- 已完成内容
- 证据或产物路径
- 当前判断
- 下一步
- 是否继续

推荐格式：

```text
【阶段确认：阶段名】
- 已完成：
- 证据 / 产物：
- 当前判断：
- 下一步：
- 是否继续：
```

### 1.3 偏差确认

如果出现下列任意情况，必须先做 `偏差确认`，不能静默改计划：

- 输入数据与预期不符
- 结果不足以支撑原计划
- 需要增加轮次、成本或范围
- 需要切换技能或下游工作流
- 发现高风险误检、误判或数据缺口

推荐格式：

```text
【偏差确认】
- 偏差来源：
- 对原计划的影响：
- 调整后的方案：
- 是否需要你确认：
```

### 1.4 收尾确认

技能结束时必须给出一次 `收尾确认`，明确：

- 最终产物
- 验证结果
- 未完成风险
- 推荐下一步或下游技能

## 2. 什么时候必须等待用户确认

以下情况必须等待用户确认，不能默认继续：

- 缺少关键输入且默认假设可能改变结论
- 要执行高成本、长时或不可逆步骤
- 要扩大检索范围、增加轮次或切换目标
- 发现结论不稳定，继续执行会误导下游

以下情况可以不等待回复，但仍要先给确认文本：

- 用户明确要求“一口气跑完”
- 当前阶段只是既定计划内的小步推进
- 没有新增成本、风险和结论歧义

## 3. 不允许出现的行为

- 没有 `计划确认` 就直接开始执行
- 阶段切换时不做 `阶段确认`
- 遇到结果偏差后静默改方案
- 明明缺输入却假装可以继续
- 在没有解释原因的情况下跳过某个工作流阶段

## 4. 技能之间的继承规则

- 上游技能调用下游技能时，必须在 `计划确认` 中说清楚“为什么调用下游技能”。
- 下游技能也必须重新输出自己的 `计划确认`，不能借用上游的计划当作替代。
- 如果多个技能串联，至少每个技能开始时一次确认、每个关键阶段一次确认。

## 5. 默认建议

- 如果用户没有额外说明，优先给“简短但完整”的 `计划确认`。
- 如果任务分三步以上，每一步结束后都做一次 `阶段确认`。
- 如果用户明确要自动推进，确认文本可以简短，但不能省略。
