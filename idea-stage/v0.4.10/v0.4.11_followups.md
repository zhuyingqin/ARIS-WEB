# v0.4.10 Codex final audit — deferred P1/P2 follow-ups

**Audit**: Codex (gpt-5.5 xhigh) on `v0.4.9..HEAD` (5 commits, +1001/-142, 10 files).
**Verdict**: READY TO SHIP.
**Action taken pre-ship**: raised MCP default timeout 60s → 300s + clamp 1..=1800s, repo `cargo fmt` on v0.4.10-touched files.

The four items below were explicitly carried to v0.4.11. Captured here so the next maintenance pass doesn't have to re-derive them from chat history.

## P1.A — Anthropic stream retry coverage narrower than OpenAI

`MessageStream::next_event` (`crates/api/src/client.rs:692`) bumps `events_emitted` on every yielded event, including `MessageStart` and empty `text_delta`s. After that, a body decode failure or premature EOF won't trigger the whole-stream restart because `events_emitted > 0`.

OpenAI executor uses a stricter "have we emitted anything meaningful?" predicate (`nothing_emitted_yet()` at `crates/aris-cli/src/openai_executor.rs:470`) which only counts non-empty content / tool fragments / reasoning fragments.

**Suggested fix**: change Anthropic side to a similar `has_emitted_meaningful_content: bool` flag, set only on first non-empty `text_delta` / `input_json_delta` / completed `tool_use`. The retry guard then matches OpenAI's behaviour and catches the "MessageStart received, then body died" failure window — currently the most common Anthropic streaming failure observed in user logs (#228-style "error decoding response body").

## P1.B — Provider-prefixed o-series models miss reasoning-effort path

`pricing_for_model()` uses `has_word()` with `-_/:` boundaries, so `openai/o3-mini` and `provider/o4` route to the right tier (`crates/runtime/src/usage.rs:125`).

But `supports_reasoning_effort()` (`crates/aris-cli/src/openai_executor.rs:37`) still uses `starts_with("o3")` / `starts_with("o4")`, so the same provider-prefixed names skip `reasoning_effort` injection and `reasoning_content` replay. Pricing and capability detection disagree.

**Suggested fix**: replace the `starts_with` chain with `has_word("o3") || has_word("o4") || has_word("o1")` (importing the helper or duplicating it locally). Also tighten the `"reasoner"` substring match — `deepseek-reasoner` is intentional, but stray third-party model names with "reasoner" in the slug would now be sent `reasoning_effort` they don't accept.

## P1.C — `stream_options.include_usage` lacks proxy compatibility fallback

`crates/aris-cli/src/openai_executor.rs:279` now unconditionally sends `stream_options: { include_usage: true }` in every streaming request body. Real OpenAI and the major OSS proxies (vLLM, SGLang, OpenRouter, Together) accept it. Less polished proxies may reject unknown top-level fields with a 400.

**Suggested fix**: detect `400 unknown field stream_options` in the OpenAI error handler and retry once without the field. The `cache_read_input_tokens` value is nice-to-have but never load-bearing, so a single graceful fallback restores compatibility without losing pricing accuracy on first-class providers.

## P1.D — Per-server MCP timeout

Even with the v0.4.10 default raised to 300s, a single workspace can mix fast (filesystem MCP, sub-second) and slow (codex agent MCP, 3-5 min p95) servers. Today both share one global `MCP_REQUEST_TIMEOUT_SECS`.

**Suggested fix**: extend `ScopedMcpServerConfig` with an optional `request_timeout_secs` field; have `McpStdioProcess` carry an `Option<Duration>` that overrides the global default. Surface this in `aris doctor` output so users can see effective timeouts per server.

## P2 — pricing substring matchers catch custom model slugs

`pricing_for_model()` matches `kimi`, `qwen`, `mimo`, `glm`, etc. via `contains()` (`crates/runtime/src/usage.rs:77`). A user-defined model called `my-kimi-clone` would now silently get Kimi pricing, which could under- or over-estimate cost.

This is an estimation-accuracy issue, not a runtime fault. Acceptable to leave until a real user reports a divergence. Possible fix: switch the same `has_word()` boundary matcher used in the o-series fix above.
